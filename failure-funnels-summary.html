ðŸŽ¯ What Do Great Evals Do?

Great evals show you which of your assumptions are False

To build a successful capability, you need to overcome a series of steps or assumptions, but the final question is:

"Did the AI succeed in the job the user hired it to do?"

Before we can assess that, there are many assumptions that if we don't validate, we have no chance of providing any value.

ðŸ’¾ Natural Language to SQL Example

Bryan breaks down SQL evaluation into sequential stages:

  1. Can it generate syntactically valid SQL?
  2. Does the query execute without errors?
  3. Does it return relevant results?
  4. Does it match user intent?
  5. Does it solve the problem?
Sequential Dependencies: If SQL doesn't run, nothing else matters. Each stage gates the next.

ðŸ”‘ Key Concepts

Relaxations

Ways to assess outcomes more flexibly. For SQL: order doesn't matter unless explicitly requested, column names can vary, formatting differences are okay.

Funnel vs Sankey

Funnel: Linear flow (Aâ†’Bâ†’Câ†’D), perfect for sequential processes
Sankey: Shows multiple paths and splits, width indicates volume

Bryan uses funnels because SQL evaluation is strictly sequential.

Binary Evaluation

Use pass/fail instead of scores. If you need a gradient, define your threshold and make it binary.

ðŸ’¡ Main Takeaways

  1. Break everything into small, testable steps
  2. Each step is a potential failure point
  3. Use transition matrices to see exactly what changed
  4. Focus on WHERE improvements happen, not just overall metrics
  5. Binary pass/fail beats complex scoring
The Core Philosophy: Don't just know it failed. Know exactly where, why, and what to fix next.
